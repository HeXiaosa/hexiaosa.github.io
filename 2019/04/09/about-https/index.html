HTTPS 加密传输

Veröffentlicht am 2019-04-09

HTTPS 相对于 HTTP 是安全的，它是加密传输的，那么它是怎么加密传输呢，数字证书又是什么，有数字证书就一定是可信任的吗，下面来一一解答

什么是数字证书

数字证书可以是组织颁发，也可以是自己颁发的，组织颁发的证书，客户端会信任该服务，直接访问，自己颁发的证书，会弹框，用户确认后才可以访问。

HTTPS 加密中使用了非对称加密，数字证书中就包含了非对称加密所需的公钥和私钥。

tip :
CA : 颁发证书的机构
CA 证书：CA 颁发的证书，就是我们说的数字证书，包括持有者身份，CA机构的签名，公钥和私钥。CA机构的签名用于验证证书的有效性。

HTTPS 是怎么加密的

对称加密是通过一个秘钥来加密和解密的，秘钥在客户端和服务端之间传输，如果被劫持，还是会被破解；
非对称加密是通过公钥加密，私钥解密的，服务端把公钥传给客户端，然后客户端用公钥加密传输的内容，传给服务端，服务端用私钥解密内容，这样就算被劫持，因为没有私钥，也无法解密。

所以使用对称加密是不安全的，非对称加密是安全的，但是非对称加密效率比较低，HTTPS 就采用了对称加密和非对称加密结合的方式。

具体过程用这篇文章里找到的这张图描述更清晰 （https://juejin.im/post/5a4f4884518825732b19a3ce）

这个过程用文字描述：
服务端把公钥给客户端，客户端在验证有效后，生成一个随机值，这个随机值就是对称加密的秘钥，公钥加密这个秘钥传给服务端，服务端用私钥解密，得到对称加密的秘钥，然后用这个秘钥加密要传输的内容。

HTTPS 证书相关

上面基本搞清楚加密传输的流程，但是加密过程中使用的数字证书，客户端是怎么验证这个证书是可信任的呢。
客户端通过 TSL 协议（SSL发展过来的，3.0改为TSL）来验证，TSL 的作用包含：

加密
证书验证
验证消息完整性

加密过程在上面已经介绍过了，这里只说下证书验证，证书验证就是从数字证书中取出颁发者，源头的颁发者非常少，于是就是看颁发者是否可信，颁发者可信，这个证书也就是可信的。
